# SPEC — SubTrack: внутренний трекер корпоративных подписок

Self-hosted веб-сервис для учёта платных подписок одной компании: какой сервис, сколько стоит, кто ответственный, кто занимает места, когда списания, план vs факт расходов по месяцам. Плюс собственный MCP-сервер и REST API, чтобы данные были доступны AI-ассистентам и скриптам (отчёты, выгрузки, вопросы вида «сколько мы тратим на дизайн-инструменты»).

Инстанс обслуживает **одну компанию** (для другой компании поднимается отдельный инстанс). Базовая валюта отчётов — **USD** (настройка инстанса, по умолчанию USD). Интерфейс — **только русский**.

---

## 1. Роли и доступ

- **Вход в UI:** Google OAuth. Вход разрешён только email'ам из белого списка (`AllowedEmail`), домен сам по себе доступа не даёт.
- Начальные администраторы задаются env-переменной `ADMIN_EMAILS` (список через запятую) — сидятся в whitelist с ролью Admin при первом старте.
- **Роли UI:**
  - **Viewer** — видит всё, ничего не меняет.
  - **Manager** — CRUD сервисов, мест, платежей, сотрудников; подтверждение списаний.
  - **Admin** — всё выше + whitelist и роли, API/MCP-токены, справочники, настройки, ручная пересборка план-снапшотов.
- **API/MCP-токены:** создаёт Admin, у токена есть имя и роль (Viewer / Manager / Admin). Хранятся в БД как хэш, значение показывается один раз при создании. Операции записи требуют роли Manager или Admin («Manager+»).

## 2. Технологический стек (зафиксирован)

- **Next.js 15 (App Router) + TypeScript (strict)** — фронт и бэк в одном приложении.
- **PostgreSQL 16 + Prisma** (все денежные поля — `Decimal`, никогда float).
- **Auth.js (NextAuth v5)**, Google provider; проверка whitelist в колбэке `signIn`.
- **Tailwind CSS + shadcn/ui**, иконки lucide-react, графики **Recharts**.
- **MCP:** официальный `@modelcontextprotocol/sdk`, Streamable HTTP транспорт на route handler `/mcp`; авторизация Bearer-токеном. REST API `/api/v1/*` с теми же операциями; MCP — тонкая обёртка над сервисным слоем.
- **Планировщик:** `croner` внутри приложения (запуск из `instrumentation.ts`, Node runtime): план-снапшот 1-го числа 00:05; обновление курсов валют ежедневно 07:00. Таймзона — env `APP_TZ`, по умолчанию `Europe/Kyiv`.
- **Курсы валют:** frankfurter.app (данные ЕЦБ), fallback — ручной ввод курса в настройках.
- **Тесты:** Vitest (unit — обязательны для всей расчётной логики), Playwright (e2e smoke).
- **Деплой:** Docker Compose (app + postgres), `prisma migrate deploy` при старте контейнера, всё за существующим reverse-proxy. Один `.env`, актуальный `.env.example` в репозитории. Готовый скрипт ежедневного `pg_dump` (хранение 30 дней).

## 3. Модель данных

Все сущности имеют `id`, `created_at`, `updated_at`. Суммы — `Decimal`.

### 3.1 AllowedEmail + User
- `AllowedEmail`: `email` (unique), `role` (Viewer/Manager/Admin), `added_by`.
- `User` — стандартные таблицы Auth.js + поле `role` (копируется из whitelist при первом входе, далее управляется Admin'ом).

### 3.2 Category (справочник, редактирует Admin)
- `name`, `color`. Примеры: Dev tools, Design, Marketing, Infra, AI, Office.

### 3.3 PaymentMethod (справочник)
- `name` («Карта \*\*\*\*1234 (Иван)», «Счёт ФОП», «PayPal team@…»), `note`, `is_archived`.

### 3.4 Service — подписка (центральная сущность)
- `name`, `vendor_url` (для автозагрузки favicon-логотипа), `category_id`, `description`
- `billing_model`: `fixed` | `per_seat` | `hybrid` (фикс-база + места)
- `billing_cycle`: `monthly` | `yearly`
- `price` — фиксированная часть (для `per_seat` = 0)
- `seat_price_default` — цена места по умолчанию (для `fixed` = null)
- `currency` (ISO 4217)
- `billing_day` (1–31) для monthly; `renewal_date` для yearly
- `next_payment_date` — вычисляется автоматически, ручная коррекция допустима
- `payment_method_id`
- `owner_id` — **ответственный, обязательное поле** (User); `backup_owner_id` — опционально
- `status`: `active` | `paused` | `cancelled` | `archived`
- `cancellation_notice_days` — за сколько дней до продления нужно решение «продлеваем/отменяем» (актуально для yearly), по умолчанию 30
- `tags[]`, `notes`

### 3.5 Employee — сотрудник (потребитель мест)
Справочник людей, на которых оформлены места. Это не пользователи трекера.
- `email` (unique), `full_name`, `department`
- `status`: `active` | `offboarded`, `offboarded_at`

### 3.6 Seat — место в сервисе
- `service_id`, `employee_id` (при добавлении по новому email сотрудник создаётся автоматически)
- `seat_price` — наследует `seat_price_default`, можно переопределить на месте
- `started_at`, `ended_at` (null = активно)
- Валидация: не более одного активного места на пару (service, employee).

### 3.7 Payment — фактическое списание
- `service_id`, `paid_at`, `amount`, `currency`
- `amount_base` — сумма в базовой валюте по курсу на `paid_at`, фиксируется при создании
- `source`: `manual` | `confirmed_expected` | `csv_import`
- `plan_line_id` — ссылка на строку плана, если платёж создан подтверждением
- `comment`, `invoice_url` (опционально)

### 3.8 PlanSnapshot + PlanLine — зафиксированный план на месяц
- Снапшот собирается автоматически 1-го числа 00:05 (cron); Admin может пересобрать вручную (действие пишется в аудит-лог).
- `PlanLine`: `service_id`, `expected_date`, `expected_amount`, `currency`, `amount_base` (по курсу даты снапшота), `breakdown` (json: фикс-часть, количество мест, сумма мест), `status`: `expected` | `confirmed` | `waived`.
- **Ключевое правило:** план месяца после снапшота задним числом не меняется. Изменения подписок влияют на «прогноз» текущего месяца и на план следующего.

### 3.9 FxRate
- `date`, `from`, `to`, `rate`. Для плана курс берётся на дату снапшота, для факта — на дату платежа. При отсутствии курса на дату — последний известный до неё.

### 3.10 AuditLog
- `entity`, `entity_id`, `actor` (пользователь или имя токена), `action`, `diff` (old → new), `ts`.
- Пишется на все изменения Service / Seat / Payment / Employee / справочников / настроек / whitelist.

## 4. Бизнес-логика

### 4.1 Стоимость сервиса
- `fixed`: `price`; `per_seat`: Σ `seat_price` активных мест; `hybrid`: `price` + Σ мест.
- **Нормализованная месячная стоимость (run-rate):** yearly → сумма / 12.

### 4.2 План (ожидаемые списания месяца M)
В план попадают сервисы `active`, у которых:
- `monthly`: списание каждый месяц в `billing_day`; если `billing_day` больше числа дней месяца — последний день месяца;
- `yearly`: месяц `renewal_date` = M.
Сумма строки = стоимость сервиса (4.1) на момент снапшота, в базовой валюте по курсу даты снапшота.

### 4.3 Факт и подтверждение списаний
- На дашборде лента «Ожидаемые списания»: кнопка ✓ превращает строку плана в Payment; в диалоге можно скорректировать сумму/дату — расхождение попадает в отчёт как дельта с пометкой.
- Ручное добавление платежа (в т.ч. вне плана — разовые доплаты).
- Строку плана можно пометить `waived` («списания не было — это нормально») с комментарием.
- Просроченным считается `expected`-строка, у которой `expected_date` + 5 дней < сегодня (константа в настройках).

### 4.4 Блок «Требует внимания» (вместо системы уведомлений)
Вычисляется на лету, показывается на дашборде:
1. Просроченные подтверждения (4.3).
2. Годовые подписки в окне решения: `renewal_date − cancellation_notice_days` ≤ сегодня ≤ `renewal_date` — «продлеваем или отменяем?».
Внешних каналов уведомлений (email/мессенджеры) в системе нет.

### 4.5 Отчёт «План / Факт» за месяц
Экран + экспорт CSV + MCP-инструмент. Содержимое:
- **Итого:** план, факт, дельта, % отклонения — в базовой валюте.
- **По сервисам:** план / факт / дельта / авто-пометка причины (новый сервис, изменение числа мест, изменение цены, курсовая разница, не подтверждено, waived).
- **Срезы:** по категориям, по ответственным.
- **События месяца:** новые подписки, отменённые, изменения цен, добавленные/закрытые места.
- **Два представления (переключатель):** кэш-флоу (годовой платёж целиком в месяц списания) и нормализованное (yearly / 12).

### 4.6 Офбординг сотрудника
Карточка сотрудника показывает активные места и их суммарную стоимость/мес. Действие «Офбординг»: чек-лист мест → массовое закрытие (`ended_at`), статус → `offboarded`, сразу видна экономия/мес.

### 4.7 Валюты
Хранение в исходной валюте + фиксация `amount_base`. В таблицах и итогах — базовая валюта; исходная сумма видна в карточке/тултипе.

## 5. UI/UX

Общее: минималистичный быстрый интерфейс, тёмная/светлая тема, русская локаль (даты DD.MM.YYYY, разделители тысяч), favicon-логотипы по `vendor_url` (с graceful fallback на инициалы), скелетоны, тосты, подтверждение разрушающих действий, undo на закрытие места, ⌘K-палитра (поиск сервисов и быстрые действия), адаптив для просмотра с телефона, осмысленные пустые состояния.

Экраны:
1. **Дашборд:** KPI-карточки (run-rate/мес; факт vs план текущего месяца с прогресс-баром; прогноз до конца месяца; активные сервисы/места), блок «Требует внимания», лента «Ожидаемые списания» с ✓, график 12 месяцев план/факт, донат по категориям, топ-5 сервисов по стоимости.
2. **Сервисы:** таблица (лого, название, категория, модель, цикл, стоимость/мес норм., мест, ответственный, следующая оплата, статус); поиск; фильтры (статус/цикл/категория/ответственный/способ оплаты); сортировка; сохранённые пресеты фильтров.
3. **Карточка сервиса:** вкладки Обзор / Места / Платежи / История (аудит-лог сущности). Добавление места — поле email с автокомплитом по сотрудникам.
4. **Сотрудники:** таблица с суммой стоимости мест на человека; карточка сотрудника; флоу офбординга.
5. **Отчёты:** выбор месяца, отчёт 4.5, экспорт CSV.
6. **Настройки (Admin):** категории, способы оплаты, базовая валюта и курсы (авто + ручной ввод), whitelist и роли, API/MCP-токены, параметры (таймзона, срок «просрочки» подтверждения).

## 6. MCP-сервер и REST API

Транспорт MCP: Streamable HTTP на `/mcp`, Bearer-токен. Ответы — компактный JSON; суммы в исходной валюте + пересчёт в базовую. REST API `/api/v1/*` повторяет те же операции.

| Инструмент | Доступ | Назначение |
|---|---|---|
| `whoami` | все | Имя и роль текущего токена |
| `overview` | все | Сводка: run-rate, факт/план месяца, ближайшие списания, «требует внимания» |
| `list_services` | все | Фильтры: status, category, owner, q |
| `get_service` | все | Полная карточка: места, ближайшие платежи |
| `create_service` / `update_service` / `set_service_archived` | Manager+ | CRUD подписок |
| `list_seats` | все | По service_id или по email сотрудника |
| `add_seat` / `end_seat` | Manager+ | Управление местами |
| `list_employees` / `get_employee_costs` | все | Справочник людей; стоимость мест человека |
| `record_payment` / `confirm_expected_payment` | Manager+ | Факт списаний |
| `get_monthly_report` | все | Полные данные отчёта план/факт за месяц (оба представления) |
| `costs_summary` | все | group_by: category \| owner \| vendor \| billing_cycle; произвольный период |
| `upcoming_payments` | все | Ожидаемые списания на N дней вперёд (по умолчанию 30) |
| `needs_attention` | все | Просроченные подтверждения + годовые в окне решения |
| `import_csv` | Manager+ | kind: services \| seats \| payments |
| `export_data` | все | kind + format: csv \| json |

## 7. Нефункциональные требования

- Объёмы: до 500 сервисов, 3 000 мест, 10 000 платежей; списки с пагинацией; отклик страниц < 500 мс на этих объёмах.
- **Unit-тесты обязательны** на расчётную логику: нормализация yearly→monthly, сборка плана (включая `billing_day = 31` в коротких месяцах и февраль), конвертация валют (в т.ч. отсутствие курса на дату), дедупликация CSV-импорта. Вся расчётная логика — чистые функции в `lib/calc/*`; UI деньги не считает.
- E2E smoke: создать сервис → добавить места → собрать снапшот → подтвердить платёж → открыть отчёт.
- Секреты только через env; токены — хэши в БД; сиды с реалистичными демо-данными для разработки.
- CSV-экспорт — UTF-8 with BOM (корректная кириллица в Excel).

## 8. Этапы

- **Этап 1 — MVP:** scaffold + Docker Compose + Prisma-схема + миграции + сиды → auth (Google + whitelist + роли) → справочники → CRUD сервисов/сотрудников/мест → модуль расчётов с unit-тестами → курсы валют → план-снапшоты + cron → подтверждение платежей → дашборд (вкл. «Требует внимания») → отчёт план/факт (оба представления) → CSV-импорт (services, seats) и экспорт.
- **Этап 2:** MCP-сервер + REST API + токены → аудит-лог (запись со старта, UI-просмотр здесь) → ⌘K-палитра → офбординг-флоу → CSV-импорт платежей.
- **Этап 3 (идеи, вне скоупа):** синхронизация сотрудников из Google Workspace Admin SDK, детект неиспользуемых мест через API вендоров.

## 9. Критерии приёмки

1. Per-seat сервис с 3 местами по $10/мес → дашборд показывает +$30/мес к run-rate; закрытие одного места → +$20/мес.
2. Годовой сервис $1 200 с продлением 15.09: нормализованный вид — $100/мес каждый месяц; кэш-флоу — $1 200 только в сентябре.
3. Место, добавленное 10-го числа, не меняет план текущего месяца (снапшот от 1-го), но видно в «прогнозе» и попадает в план следующего месяца.
4. Подтверждение ожидаемого списания в один клик создаёт Payment и переводит строку плана в `confirmed`; скорректированная сумма даёт дельту в отчёте с пометкой.
5. Сервис в EUR при базовой USD конвертируется по курсу даты снапшота (план) и даты платежа (факт); курсовая разница отображается как причина дельты.
6. Отчёт сходится: сумма по сервисам = итогу; CSV открывается в Excel без битой кириллицы.
7. `billing_day = 31` в феврале даёт ожидаемое списание 28/29 числа.
8. Вход с Google-аккаунтом вне whitelist — отказ с понятным сообщением; добавление email в whitelist открывает доступ без перезапуска.
9. MCP: `whoami` работает с любым валидным токеном; `create_service` с Viewer-токеном → 403, с Manager — успех.
10. Офбординг сотрудника с 4 местами закрывает их одним действием и показывает экономию/мес.
11. Любое изменение цены сервиса видно в истории с автором и old → new.
